## 在不确定中稳定产出——用假设树 + MVP Gate 驱动科研闭环

- **痛点（1 句）**：探索性研究容易散落——实验跑了很多，但结论不可追溯、决策无依据、知识无法复用。
- **解法一句话**：$\underbrace{\text{Hub 假设树}}_{\text{拆问题}}\;\xrightarrow{\text{MVP + Gate}}\;\underbrace{\text{可交付结论 + 设计原则}}_{\text{沉淀知识}}$
- **核心价值**：让任何人都能追溯"为什么走这条路"，在不确定任务中稳定迭代到结果。

| 组件 | 符号 | 职责 | 产出 |
|------|------|------|------|
| 🧠 Hub | $\mathcal{H}$ | 问题拆解 → 假设树 → 战略导航 | 共识表、设计原则 |
| 🗺️ Roadmap | $\mathcal{R}$ | 假设 → MVP + 验收阈值 + 止损规则 | 实验进度、数值结果 |
| 📗 Exp | $\mathcal{E}$ | 单实验标准化报告 | 关键数字、图表、洞见 |
| 🚦 Gate | $\mathcal{G}$ | 量化决策：加数据？加模型？换方向？ | Go/No-Go 决策 |

```
信息流闭环:
GPT 脑暴 → session.md → hub.md (假设树)
                              ↓
               roadmap.md (MVP + Gate)
                              ↓
                     exp.md (实验报告)
                              ↓
           结论 + 关键数字 → 回流 hub.md
                              ↓
                 沉淀 → 设计原则 + 知识卡片
```

核心痛点是什么呢？做研究的时候实验跑了很多，但经常出现：结论散落在各处、决策缺乏依据、下次再做类似问题又得重来一遍。

我的解法是一套闭环系统：先用 Hub 的假设树把大问题拆成可证伪的小假设，再用 Roadmap 把每个假设变成一个最小可行实验 MVP，提前写好验收阈值和止损规则。跑完实验产出标准化报告，关键数字自动回流到 Hub，最后沉淀成可复用的设计原则。

这样做的价值是：在不确定性很强的任务里，我能稳定地推进到可交付的结果，而且团队里任何人都能追溯"为什么走这条路"。

## 三层架构 + 信息回流：假设树 → MVP → 标准化实验 → 知识沉淀

- **架构设计**：分层解耦"战略思考"与"执行追踪"。

```
┌─────────────────────────────────────────────────────────────────┐
│                 Layer 0: Master Hub (全局战略)                    │
├─────────────────────────────────────────────────────────────────┤
│     Layer 1: Topic Hubs (研究主题)                                │
│     [scaling/]  [moe/]  [vit/]  [benchmark/]                    │
├─────────────────────────────────────────────────────────────────┤
│     Layer 2: Experiments (单实验报告)                             │
│     exp_*.md + img/ + 数值结果                                    │
├─────────────────────────────────────────────────────────────────┤
│     Layer 3: Knowledge Artifacts (知识沉淀)                       │
│     card/*.md + design/principles.md                            │
└─────────────────────────────────────────────────────────────────┘
```

- **MVP 设计三要素**：

| 要素 | 说明 | 示例 |
|------|------|------|
| 精确假设 | 一句话可证伪 | "高噪声下 MoE 比全局模型 ΔR² ≥ 0.05" |
| 最小配置 | 只改一个变量 | 固定数据/模型，只变 routing 策略 |
| 量化验收 | If-Then 规则 | If ΔR² ≥ 0.05 → 接受; Else → 拒绝并关闭 |

- **Decision Gate（资源分配决策）**：

```
Q: 下一步该投什么？
│
├── Gate-1: 理论上限 (Fisher/CRLB)
│   └── If headroom > 20% → 继续投模型
│
├── Gate-2: Scaling 趋势
│   └── If Δ < 5% per 3× data → 不投数据
│
└── Gate-3: 结构红利 (Oracle)
    └── If ΔR² > 0.05 → 投结构 (MoE/分域)
```

<details>
<summary><b>note</b></summary>

这套系统的核心是分层架构和信息回流。

分层上：Hub 负责战略思考——假设树、洞见汇合、设计原则；Roadmap 负责执行追踪——MVP 列表、进度、数值结果。两者职责清晰：Hub 回答"我们知道了什么、往哪走"，Roadmap 回答"计划跑哪些实验、进度如何"。

MVP 设计有三个硬性要求：假设必须可证伪、配置必须最小化只改一个变量、验收必须量化写成 If-Then 规则。

Decision Gate 是资源分配的核心机制——我用三个信号做裁决：理论上限还有多少空间、Scaling 趋势是否饱和、Oracle 实验验证结构红利是否存在。这三个信号共同决定"该加数据还是加模型还是换方向"。

</details>

## 核心价值：可追溯 + 可复用 + 可迭代

- **核心结论**：「假设驱动 + 量化 Gate」让探索性研究变得可控——每个决策都有证据链支撑。
  - 机制层：问题拆解成假设树 → MVP 最小验证 → Gate 量化决策 → 洞见提炼 → 下一轮迭代
  - 证据层：所有结论都可追溯到具体实验报告、图表、数值

- **系统解决的 3 个问题**：

| 问题 | 传统做法 | 本系统 |
|------|---------|--------|
| 结论不可追溯 | 口头讨论、散落笔记 | Hub 假设树 + Exp 标准化报告 |
| 决策无依据 | "感觉应该这样" | Gate 量化阈值 + 止损规则 |
| 知识不复用 | 每次从零开始 | 设计原则 + 知识卡片自动沉淀 |

- **Cursor AI 集成**：快捷命令驱动工作流

| 命令 | 作用 |
|------|------|
| `n` | 新建实验（创建 hub/roadmap/exp） |
| `a` | 归档实验（标准化报告 + 同步 hub） |
| `u` | 更新文档（洞见回流 + git push） |
| `?` | 查看进度状态 |

- **下一步（使用本模板）**：
  1. 🟢 Fork/Clone 此模板，配置 `.cursorrules` 中的项目信息
  2. 🟢 创建第一个主题 `experiments/[topic]/`，用 `n` 命令立项
  3. 🟡 跑实验、用 `a` 归档、让知识自动沉淀
